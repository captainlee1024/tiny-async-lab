# 导读

本书沿着“语言契约 → 成熟运行时 → 底层 I/O → 自研实现 → 工程实践 → 专用变体”的因果链，系统解释 Rust 异步程序如何运行，以及异步运行时为什么需要相应的抽象与实现。

目标不是尽快拼装一个可运行的 executor，而是在固定版本的公开契约、源码和可复现实验之间建立能够相互验证的理解。

## 全书结构

| 部分 | 核心问题 |
| --- | --- |
| Rust 异步契约与编译器模型 | `Future`、`Poll`、`Waker`、`Pin` 和 `async`/`.await` 如何共同表达异步计算？ |
| Tokio 与 Mio | 成熟运行时如何组织任务、调度、时间和 I/O，readiness 又如何从操作系统传递到任务？ |
| tiny-mio | 如何从非阻塞 I/O 和平台事件机制开始，建立独立于运行时的最小 I/O 抽象？ |
| 通用 tiny-runtime 基线 | 如何实现 executor、任务唤醒、timer、reactor、取消和关闭，并说明它们的不变量？ |
| 异步工程实践 | Tokio 与自研运行时如何分别实现同一组真实场景，并处理背压、阻塞边界和资源生命周期？ |
| 专用运行时变体 | 如何从真实 workload 提出可证伪的性能假设，验证专用实现，并解释收益、退化与适用边界？ |

章节只在对应研究或实现开始时创建；内容变大时优先拆分章节与子章节，而不是提前拆成多本相互依赖的书。

## 阅读方式

第一次阅读建议按照目录顺序推进，因为后续实现会直接依赖前面建立的语言契约和状态模型。

每个机制章节将区分公开保证、固定版本实现、设计原因和实验观察，并链接到对应的可运行实验或项目实现。

希望先观察行为的读者可以从章节链接的实验进入，再回到正文理解产生该行为的契约与实现。

## 内容边界

本书保存已经验证的机制解释，不把未经验证的文章、个人笔记或暂时假设写成确定结论。

公共 API 的使用契约保留在 crate rustdoc，可运行证据保留在 `labs/`，持续演进的实现保留在 `crates/`，场景代码保留在 `practices/`。
