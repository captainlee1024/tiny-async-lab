# 源码阅读方法

本项目同时研究异步系统如何工作，以及成熟 Rust 项目如何组织高质量代码。这两条学习线并行推进，但不在同一遍阅读中争夺注意力。

## 为什么分成两遍

优雅代码通常只是约束条件的外在结果，而这些约束很容易被忽略，例如公共 API 兼容性、pinning、取消、平台差异、调度器竞争、unsafe 不变量、编译时间和 feature 组合。如果还没有还原这些约束就开始评价代码风格，很容易只模仿形式，却不理解设计目的。

因此，每个机制单元先完成一遍即时的原理阅读，再在里程碑阶段完成一遍工程质量复盘。

## 理解闭环与研究边界

“不留下黑盒”是指当前结论不依赖未命名、未解释且没有回收计划的机制，不是无边界地追溯到 LLVM、操作系统、CPU 或硬件。
每个机制单元需要闭合以下五层：

| 层次 | 必须回答的问题 | 主要证据 |
| --- | --- | --- |
| 契约 | 调用者与实现者分别可以依赖什么、必须保证什么 | 官方语言与 API 文档、契约测试 |
| 行为 | 正确、失败、取消和资源释放如何被观察 | 聚焦且可复现的实验 |
| 实现 | 固定版本中经过哪些状态、控制流和 symbol | 固定源码、内部注释与测试 |
| 原因 | 为什么选择当前设计，替代方案及代价是什么 | RFC、设计文档、PR、历史与维护者说明 |
| 工程 | 模块、类型、测试、unsafe 和平台边界如何保护约束 | 第二遍源码阅读与本项目对照实验 |

闭合不要求把未知伪装成答案；证据仍不足时，应记录已经核验的范围、尚缺的证据和复查条件。
学习顺序允许暂时前向引用尚未展开的概念，但必须写明缺少什么、为何不阻塞当前结论，以及它在 `ROADMAP.md` 或 `research/TOPICS.md` 中的回收位置。
如果缺失概念会改变当前结论，就不能暂缓；确实超出项目范围的边界也要说明停止位置和理由。

## 学习单元与章节拆分

`ROADMAP.md` 保存阶段、机制依赖和完成状态，`docs/src/SUMMARY.md` 保存实际阅读顺序，`research/TOPICS.md` 保存尚未闭合的问题和核验目标。
只细化即将进入的阶段；新信息出现后及时调整当前阶段，不为远期阶段预建大量任务或空章节。

出现以下任一情况时，应考虑拆出子章节或独立实验：

- 开始回答另一个可以独立验收的主要问题；
- 出现独立的契约、状态机、所有权、生命周期、安全或平台边界；
- 契约、实现或设计原因需要沿不同证据路径核验；
- 可以用独立实验验证而不依赖当前章节的其余内容。

不要只因字数增加机械拆成上下篇，也不要把同一条推理链分散到多个位置。
父章节保留问题地图和依赖关系，子章节各自闭合一个主要问题。

## 第一遍：语义与正确性

优先回答以下问题：

1. API 或内部组件承诺了什么契约？
2. 它保存了哪些状态，允许发生哪些状态转换？
3. 哪些值可以移动、共享或 drop，哪些值必须活过一次 poll 调用？
4. 谁负责推动进度，什么事件会触发下一次 poll 或事件处理？
5. 遇到 `Pending`、取消、关闭、panic 和部分初始化时会发生什么？
6. 每一处 unsafe 操作分别由哪些不变量保证安全？
7. 哪些行为属于正确性要求，哪些只是性能策略？
8. 当前观察适用于哪个 target、哪些 feature flags 和哪个上游 commit？

这一遍应产出机制笔记；有必要时补充一张小图，并建立一个聚焦、可执行的实验。

## 同步研究设计动机

每个机制单元先用当前契约和最小实验确定问题，再从 `research/TOPICS.md` 选择直接相关的设计资料，包括按主题登记的 Without Boats 与 Niko Matsakis 的文章或访谈。
阅读笔记必须区分来源陈述、作者的设计判断和本项目的推导，并用 RFC、tracking issue、PR、Git 历史或固定源码核验仍影响当前结论的主张。
没有找到足够的设计原因证据时，明确记录未知，不根据作者声誉或实现外观补写动机。

## 即时记录代码设计

第一遍阅读时可以记录看起来值得复查的设计，但不要立刻把它判断为普遍适用的“最佳实践”。例如：让非法状态无法表示的类型、范围很小的 unsafe 边界、异常清晰的命名，或者能够暴露隐藏不变量的测试辅助工具。

每条记录至少包含：

- 源码仓库、commit、文件路径和 symbol；
- 它看起来要解决的约束；
- 第二遍阅读时需要验证的问题。

## 第二遍：工程设计与代码质量

理解机制单元后，再回头回答以下问题：

1. 为什么这里使用 module、type、trait、method 或 free function？
2. 前置条件、检查和使用能否在局部完成推理，还是必须跨越多个函数或类型追踪？
3. helper 或 context type 保护了什么不变量，还是只把单次逻辑移动到了另一个位置？
4. 哪些不变量由类型表达，哪些仍依靠注释或运行时检查？
5. 命名和源码顺序如何体现所有权、生命周期、readiness、控制流和数据流方向？
6. 公共 API、内部策略、平台代码与 unsafe 代码如何隔离？
7. 错误如何分类、补充上下文、转换，或者被有意隐藏？
8. feature flags 和条件编译如何避免污染主路径？
9. 测试保护的是什么：示例、契约、状态转换、竞态，还是回归问题？
10. 哪些复杂度来自兼容性或规模，不应该复制到小型实现中？

这一遍应产出简短的设计复盘，而不是“聪明语法”的陈列清单。

## 设计模式记录模板

可复用的观察使用以下模板：

```text
模式：
来源：
上下文与约束：
实现机制：
为什么有效：
代价与取舍：
不适用的情况：
实验或证据：
tiny-async-lab 的决定：
```

## 如何应用一种上游模式

把上游设计用于本项目代码前：

1. 在小型实验中复现它要解决的问题。
2. 先实现本地最简单的正确方案。
3. 在相同约束下与上游模式比较。
4. 明确选择采用、简化或拒绝该模式。
5. 为促成该决定的不变量添加测试。

这样既能避免逐行仿写，也能真正学习标准库、Tokio 和 Mio 的高质量设计。

## 执行节奏

代码质量学习在小范围内同步进行，在深入程度上适当延后：

- 第一遍阅读期间，立即记录值得复查的设计决策。
- 每完成一个机制单元，用较短的第二遍阅读复盘工程取舍。
- 每到达一个子系统里程碑，跨越模块边界完成一次深入复盘。
- 在定稿对应的自研子系统前，对比上游设计、本项目当前需求与已记录约束。
