# ExecPlan 规范

ExecPlan 是复杂任务的可恢复执行记录，使没有原对话的新 agent 能够从仓库和工作树继续任务。
它不替代 `ROADMAP.md`、ADR、正式技术文档、Git 历史或 PR 正文。

## 启用条件

满足以下任一条件时必须使用 ExecPlan：

- 预计跨越多个 agent 会话或多个 PR；
- 属于 L3 变更；
- 涉及跨组件依赖、公共 API、unsafe、平台边界或重要并发不变量；
- 存在需要原型或实验消除的显著未知；
- 需要跨机器恢复或交接给没有原上下文的 agent；
- 包含多个能够独立验收的里程碑。

目标明确、能够在一次会话中完成并验证的 L1 变更不创建 ExecPlan，普通命令、微小文档修正或机械修改也不增加计划文件。

## 存放与生命周期

计划使用以下路径：

```text
docs/agent-workflow/plans/
├── active/
│   └── <short-kebab-description>.md
└── completed/
    └── <short-kebab-description>.md
```

没有真实计划时不创建空目录或占位文件；文件名描述任务目标，不加入 agent 名称、会话 ID 或 `final-v2`。

任务开始前创建 active plan，并在每个里程碑完成、重要决定形成、意外发现出现和暂停前更新。
全部验收条件满足后，补齐结果与复盘再移动到 `completed/`；completed plan 是历史执行记录，其中仍然有效的结论必须已经提升到当前权威位置。

任务被替代时，在原计划中写明原因和替代计划路径，再移入 `completed/`，不要保留两个声称权威的 active plan。

## 写作约束

- 假设读者没有原对话，但可以按照链接读取仓库文件。
- 链接稳定事实源，不复制会独立演化的规范、架构或路线图正文。
- 使用准确的仓库相对路径和 symbol；行号只能辅助定位。
- 进度项包含日期和可观察证据，不把计划执行的动作写成完成事实。
- `Decision Log` 记录影响后续执行的局部决定；持久且难以撤销的决定仍进入 ADR。
- `Surprises and Discoveries` 区分已验证发现与待验证问题。
- 命令写明工作目录、关键参数和预期观察结果。
- 恢复步骤必须安全且可重复；潜在破坏性步骤说明回退或重新开始方法。
- 计划保持可独立恢复所需的信息密度，不保存完整 transcript 或大段工具输出。

## 最小模板

````markdown
# <任务结果>

- 状态：active
- 最近更新：YYYY-MM-DD
- 关联路线图：`ROADMAP.md` 中的具体阶段或条目
- 关联 PR：无 / URL

## 目标

说明完成后可以观察到的结果，以及为什么现在需要它。

## 本次不做

明确排除容易造成范围扩张的相邻工作。

## 上下文地图

- `path` — 为什么恢复任务时必须查看它。
- `path::symbol` — 当前相关契约或不变量。

## 验收条件

- [ ] 可观察结果及其验证方式。

## 里程碑

### 1. <独立结果>

说明预期改动、验证和完成标志，不预先展开未经证据支持的实现细节。

## Progress

- [ ] `YYYY-MM-DD` — 当前事实、证据和剩余工作。

## Surprises and Discoveries

- 已验证：结论、证据和对计划的影响。
- 待验证：问题、缺失证据和下一项实验。

## Decision Log

- `YYYY-MM-DD` — 决定、理由、替代方案和影响范围。

## Validation and Acceptance

- `command` — 实际结果或尚未执行。

## Idempotence and Recovery

说明如何安全重试、恢复失败步骤，以及哪些本地状态不能跨机器保留。

## Next Step

只写当前最准确的一个下一步，包含路径、命令或待回答问题。

## Outcomes and Retrospective

任务完成前保留为空；完成时总结结果、偏差、遗留问题和需要提升到正式文档的知识。
````

模板 section 可以按任务删减无关内容，但不得删除目标、非目标、验收、进度、验证和下一步。

## 恢复核验

新 agent 恢复计划后，必须先回答以下问题再修改文件：

1. 整个任务的结果和非目标是什么？
2. 当前里程碑是什么？
3. 哪些完成声明已由 Git、源码或检查结果证实？
4. 哪些声明已经过期或仍缺少证据？
5. 当前准确的下一步和最小验证是什么？

计划无法回答这些问题时，应先补足计划或链接的权威上下文，而不是依靠猜测继续。

## 完成核验

完成计划前必须：

1. 核对全部验收条件，而不是只核对最后一个里程碑。
2. 运行与风险相称的检查，并记录实际结果。
3. 将长期结论提升到正式文档、rustdoc、ADR 或 `ROADMAP.md`。
4. 明确未完成事项属于缺陷、后续任务还是主动非目标。
5. 填写结果与复盘并移入 `completed/`。

L3、unsafe、关键并发正确性和重要性能结论应使用新的只读 review task 进行独立核验；是否增加独立 evaluator 自动化由实际收益决定，不作为所有任务的固定成本。
